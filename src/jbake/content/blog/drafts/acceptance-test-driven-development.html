I recently got the opportunity to showcase our development environment off to our client.  Our team was rather proud of our setup, which centers on our continuous integration server, Hudson.  Hudson will execute a build after each check-in, run all unit tests, run all integration tests, deploy to an development server, run performance tests, run a suite of static code analysis tools (CheckStyle, FindBugs, PMD, Cobertura), update JIRA and lastly update our Nexus repository.  If you are currently doing continuous integration, then that may not seem impressive, but you would be amazed at how many teams don't do this.  During the course of the presentation, the question was asked, "Well this is great, all the unit and integration testing is great, but how do I know the code you wrote is satisfying a business requirement".  It was a great question and a gap in our current process, one which we needed to rectify.

In the past, the way would have handled this would be to call up IBM, tell them we want to purchase the Rational suite so that we can provide complete traceability between requirements, design, code, and test.  Well, if anybody is currently doing this, you already know how cumbersome those tools are and how they never seem to deliver on any promises.  I knew we needed something that was fast, lightweight, simple to implement and easy to maintain over time.  It had to provide a non-technical view into our requirements while providing traceability to test cases that validate our code.  I did some hunting on the internet and came across an approach that I thought could solve our problem, Acceptance Test Driven Development (ATDD).

What is ATDD?

The easiest way to look at ATDD is to think of your requirements as an executable test.  If you are familiar with or have done any Test Driven Development, then you merely need to take the same concept and scale it up to a system level.  In TDD, developers write unit tests first, that define the expected behavior of piece of code they are about to write.  Once the behavior is defined (and the test fails), the developer then goes about writing the code to satisfy the test.  His goal is to make the test pass in the simplest approach possible.  Once the test has passed, then the developer can go back and refactor the code with confidence since he now has a test to ensure the behavior of the class doesn't change.
