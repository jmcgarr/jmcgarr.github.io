Break your App into Separate Components

16 August 2011

http://googletesting.blogspot.com/2011/06/testing-at-speed-and-scale-of-google.html

Discuss why large componentized apps should break up component lifecycles

------Draft Email

It is more a question of code organization.  If I have an application that is comprised of two components and a web application, and I change one of the components, I should only have to run the build for the component I changed and the web application.  (assuming the two components aren't related)  If WebApp depends on ComponentA which in turn depends on ComponentB, and I change ComponentB, then I must build all three.
To clarify your point, if you build a component/app, you should run all it's tests for every build.  Breaking your application into seperate components with seperate build/release lifecycles can help speed/optimize your build time. 
It also can ensure that a change to a component isn't integrated until that change is fully vetted.  If I change ComponentB, but the change breaks the build, then ComponentA and the WebApp won't run their builds, which means they won't integrate with the broken code, which means in theory, they can still be deployed without waiting for ComponentB to be fixed (assuming their builds pass). 
The disadvantage is that unless you are smart about managing your source tree, you might have to check out multiple projects from source control seperately, whereas before in a large application, you had one checkout/checkin.
